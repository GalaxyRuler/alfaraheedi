## Release Decision

Decision: Desktop-foundation Phase 10 PR is ready from the local/VM QA evidence lane, with browser-store and Office sideload/live-editor coverage deferred.

Blocking issues:

- None for the narrowed desktop-foundation evidence lane after the 2026-06-30 Agent-Dev-01 packaged-app apply/copy/replace and no-selection hotkey proofs.

Deferred integration gates:

- Browser extension local-ready artifacts and public privacy URL readiness pass, but public browser-store readiness remains deferred until live production-editor browser QA, manual screen-reader review, and Chrome Web Store / Edge Add-ons dashboard steps have account-side evidence.
- Office add-ins remain a packaged task-pane foundation until Word selected-text read/apply, live PowerPoint sideload QA, and task-pane accessibility/keyboard QA are completed.
- VM automation used PowerShell Direct with a temporary host-side encrypted credential file that was deleted after each run.

Remaining risks:

- The RC installer has been rebuilt and checksum-verified locally; VM acceptance installed the copied setup executable with SHA256 `60C080871ED14F96348CB58587F6ED3713864B59F18195F7E769B887C8966D8C`, not direct `target/` output.
- A backend VM launch regression was found and fixed: when started from a non-interactive window station, the global-shortcut plugin previously caused a startup panic (`This operation requires an interactive window station`, exit code 101). The current installed setup executable stays alive after launch with no stderr in that backend context.
- The installer is unsigned unless the signing credentials from the release-signing plan are available in the release environment.
- Office and browser integration QA must avoid private documents, account data, and raw selected text in public artifacts.
- The installed app binary is still named `alfaraheedi.exe` inside the `Nahou` install directory; user-facing registry and shortcuts show `Nahou`.
- The user PC still has a pre-existing `Alfaraheedi 0.5.0` install and `Alfaraheedi.lnk` shortcut; these were not removed without explicit destructive-action approval.
- The VM desktop lane now proves packaged-app Notepad hotkey capture, apply-safe, corrected-copy, direct replacement, and no-selection behavior. On the user PC, explicit installed-app Notepad capture works. Word can discover and launch the Nahou Office task pane from the shared-folder catalog, but selected-text read/apply remains unproven and deferred.

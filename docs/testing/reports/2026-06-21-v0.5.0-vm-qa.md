Date: 2026-06-21

This pass tested the v0.5.0 Windows release artifacts inside a local Hyper-V
Windows QA VM.

Artifacts tested:

- Installer: `target\release\bundle\nsis\Alfaraheedi-0.5.0-windows-x64-setup.exe`
- Developer zip: `dist\alfaraheedi-v0.5.0-windows-x64.zip`

Expected hashes:

- Installer SHA256: `0145F22565D36AC4DA559E1B8C60C5E736F865B22CD2884FAF5AEB786933BBC9`
- Developer zip SHA256: `2DE9695D750A2FBC39FF02906BDB128D2F81685301136861DC9355562FA28939`

## VM Environment

- VM: local Hyper-V Windows QA VM
- Guest: local non-admin QA user
- Access path: Hyper-V PowerShell Direct
- Desktop state: existing disconnected desktop session
- Checkpoint action: no checkpoint created or restored during this pass

The disconnected desktop is enough for install, package, process, API, and UI
Automation inspection. It is not a reliable substitute for a live interactive
Windows session when testing global hotkeys and keyboard-driven copy/paste.

## Packaged API Smoke

Result: Pass.

The API smoke started `write-api.exe` from the extracted developer zip inside
the VM, not from the source tree.

Verified:

- `GET /healthz`
- `GET /v1/health`
- `GET /v1/rules`
- `POST /v1/analyze` with English writing mode
- `POST /v1/apply` with English writing mode
- `POST /v1/analyze` with Arabic writing mode
- `POST /v1/apply` with Arabic writing mode
- `GET /v1/llm/status` returns unavailable when no local LLM runtime is configured

The Arabic API check used raw .NET HTTP bytes and explicit UTF-8 decoding because
Windows PowerShell 5 can corrupt Arabic JSON request or response text when using
its default `Invoke-RestMethod` string encoding.

## Not Verified In This VM Run

- Global hotkey capture from another app.
- Keyboard-driven copy/paste replacement into the source app.
- Screenshot capture.
- Word, PowerPoint, WhatsApp, and browser field behavior.
- Real local LLM runtime behavior.

The hotkey/capture automation ran from a disconnected scheduled-task desktop.
The app opened and UI Automation could inspect it, but the capture attempt ended
with the visible UI message `Could not send keyboard shortcut.` Screenshot
capture also failed with an invalid screen handle. This is a limitation of this
VM automation mode for input-desktop testing, so live interactive QA on the
user PC remains the release gate for hotkey and replacement behavior.

## Final Assessment

No release-blocking issue was found in the installer, developer zip, installed
desktop launch, embedded rules, CLI safe-fix path, or packaged API behavior.

The only unresolved VM evidence is the cross-app hotkey/copy/paste loop, which
was not reliably testable from the disconnected VM session. The user's manual
host-PC QA remains the relevant evidence for that behavior.

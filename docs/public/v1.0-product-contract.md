# Nahou v1.0 Product Contract

Nahou v1.0 is a Windows-first, local-first writing companion foundation.
It ships the desktop installer as the primary user artifact and keeps browser
and Office integrations as separately gated foundations until their live
manual QA and store/sideload evidence are complete.

## Supported Primary Path

1. Install Nahou with the Windows setup installer.
2. Select text in a supported app.
3. Press the configured hotkey, default `Ctrl+Alt+A`.
4. Review deterministic local suggestions.
5. Copy corrected text or replace the original selection when the guarded
   apply path is available.

The desktop installer is the primary v1.0 product path. CLI, local web, browser
extension, and Office add-in packages are supporting or optional paths unless a
release note says otherwise.

## Release-Blocking v1.0 Surface

| Surface | v1.0 support statement |
| --- | --- |
| Notepad selected text | Desktop hotkey capture, local review, deterministic suggestions, and a guarded copy-corrected-text or clipboard-paste apply path after manual confirmation. |
| No selection or unsupported selection | The desktop app shows a clear no-selected-text or unsupported-app state without logging raw text. |

## Foundation Integrations

These packages may be built and validated as local-ready artifacts in v1.0, but
they are not public release blockers for the desktop foundation unless a release
note explicitly includes that integration.

| Surface | v1.0 foundation statement |
| --- | --- |
| Browser extension | Local-ready package and store-submission bundle may be produced. Public browser-store release remains gated by live production-editor QA, manual assistive-technology review, and store-dashboard completion. |
| Office add-ins | Word and PowerPoint task-pane foundation may be packaged. Sideload readiness remains gated by live Word/PowerPoint selected-text read/apply QA and task-pane accessibility review. |
| WhatsApp Web, Gmail, Google Docs, rich web editors | Future integration targets. Do not claim release-complete support until disposable live-editor QA passes. |
| Word and PowerPoint selected text | Future desktop and Office integration targets beyond the foundation. Do not claim document-native read/apply support until live sideload QA passes. |

## Quality Thresholds

- Zero known crashes in supported surfaces at release time.
- No raw selected text in logs, reports, screenshots, public artifacts, or telemetry.
- Deterministic safe fixes pass the release eval with zero known false positives
  in release-blocking fixtures.
- The normal Windows user path starts from the installer and does not require
  PowerShell.
- Clipboard restore succeeds in normal text-clipboard cases, and failures are
  visible to the user.

## Rule-Based And Optional LLM Scope

Nahou deterministic safe fixes are rule-based and local. They are intentionally
narrow and must be backed by positive and negative fixtures before they can be
eligible for safe auto-apply.

Optional LLM suggestions are a second-pass assistant. They require explicit
local runtime configuration, do not use bundled model weights, and are not part
of the deterministic safe-fix path.

## Explicit Non-Goals

- No hosted text processing.
- No telemetry.
- No raw selected text logs.
- No bundled LLM model weights.
- No universal live underline overlay across every desktop app.
- No guarantee of complete Arabic morphology or full English grammar.
- No UI Automation replacement until a separate real-app pilot proves it safe.
- No store approval claim for browser extension, Microsoft Store, or AppSource
  until the external review is complete.

## Current Limitations

- UI Automation text capture is best effort and depends on the target
  application's accessibility provider.
- Replacement remains guarded and may fall back to copy-corrected-text when the
  source selection is stale or unsupported.
- Browser extension and Office add-in packages require their own manual release
  gates before public store submission or sideload-readiness claims.
- Google Docs and rich web editors may expose custom editing surfaces that are
  unsupported or only partially supported.
- Local LLM quality, latency, and privacy depend on the user-controlled local
  runtime.

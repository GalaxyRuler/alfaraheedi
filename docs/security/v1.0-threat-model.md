# Nahou v1.0 Threat Model

Last updated: 2026-06-25

Nahou v1.0 is a Windows-first, local-first writing companion. The primary trust
boundary is the user's machine: selected text should be processed locally by the
desktop app, local API, browser extension bridge, Office add-ins, and optional
user-controlled local LLM runtime. Nahou does not operate a hosted
text-processing service for v1.0.

## Desktop Hotkey

Surface: global hotkey captures selected text from the active app and opens the
desktop review flow.

Risks:

- hotkey triggers in an unintended app or field;
- selected text from a sensitive context enters the review flow;
- diagnostics accidentally include selected text;
- replacement action targets a stale or unintended surface.

Controls:

- first-run privacy screen describes local processing and no text retention;
- large selections are refused by default;
- diagnostics record capture method and error category, not raw selected text;
- replacement remains clipboard-paste based for v1.0 until UI Automation
  replacement is proven;
- public claims are limited to supported surfaces and best-effort capture.

## Clipboard Capture And Restore

Surface: clipboard is used to capture selected text and restore prior clipboard
state where normal text clipboard cases allow it.

Risks:

- prior clipboard contents are lost;
- binary or non-text clipboard formats are mishandled;
- captured text appears in logs or QA reports;
- another process changes the clipboard during the flow.

Controls:

- normal text clipboard restore is a release acceptance criterion;
- unsupported clipboard states show user-facing notices;
- raw selected text is forbidden in logs, public artifacts, and source-controlled
  QA reports;
- manual QA includes no selection, binary clipboard, and restore cases.

## UIA Capture

Surface: Windows UI Automation can expose selected text for supported native
controls.

Risks:

- provider support is incomplete or app-specific;
- sensitive controls may expose text through provider metadata;
- app focus or selection can change between capture and review;
- universal desktop support could be overstated publicly.

Controls:

- UIA remains capture-first and replacement is not claimed for v1.0;
- fallback behavior is documented;
- public docs describe supported surfaces and limitations;
- sensitive-field protection is not claimed as universal for UIA.

## Browser Content Scripts

Surface: Manifest V3 extension detects supported editable fields and sends active
field text to the configured loopback Nahou API.

Risks:

- content script reads text from a field that looks editable but is sensitive;
- page scripts or rich editors use hidden sentinels or nested editor internals;
- local API URL is changed by page-origin messages;
- extension package includes remote code or unnecessary permissions.

Controls:

- permissions are minimized and documented in store materials;
- local API URL must be loopback;
- content-script messages cannot override stored API URL or writing mode;
- password, payment, token, API-key, one-time-code, read-only, disabled, hidden,
  and sensitive-looking fields are excluded on a best-effort basis;
- remote code, `eval`, `new Function`, and remote imports are forbidden in the
  release package.

## Office Add-ins

Surface: Word and PowerPoint task panes read selected document text and apply
guarded replacement where Office APIs permit it.

Risks:

- broad manifest host or requirement claims exceed tested behavior;
- selected range changes between analysis and replacement;
- production manifest points at localhost or missing public privacy/support URLs;
- Office document content is accidentally stored in reports or logs.

Controls:

- dev and production manifests are separated;
- production manifest uses public HTTPS privacy/support URLs;
- replacement is guarded by re-reading selection where the API permits;
- stale or unsupported selection states are surfaced to the user;
- manual Word and PowerPoint sideload QA is required for release gates.

## Local API

Surface: local service provides deterministic analysis and optional local LLM
suggestion endpoints.

Risks:

- API binds beyond loopback;
- request or response bodies are logged;
- local integrations send text to an unexpected endpoint;
- error reports contain raw user text.

Controls:

- integrations target loopback local API endpoints;
- no hosted fallback is provided;
- raw request text is not logged;
- public release hygiene scans source-controlled reports/log artifact locations
  for obvious selected-text samples.

## Local LLM Runtime

Surface: optional second-pass suggestions can call a user-configured local
OpenAI-compatible runtime.

Risks:

- user points Nahou at a runtime they do not control;
- model output is malformed, unsafe, or overbroad;
- LLM path becomes confused with deterministic safe fixes;
- local runtime network behavior is outside Nahou's control.

Controls:

- LLM use is disabled until configured and consented;
- consent copy warns that selected text is sent to the configured local runtime;
- strict JSON schema validation rejects malformed or oversized suggestions;
- deterministic safe fixes remain the primary release path;
- ONNX Runtime GenAI and embedded model weights remain future review items.

## Release Artifacts

Surface: installers, browser extension packages, Office add-in packages, release
notes, public docs, screenshots, and QA evidence.

Risks:

- public artifacts include private paths, internal agent files, credentials, or
  raw selected text;
- browser/Office packages make privacy claims that do not match behavior;
- dependency/license changes introduce release risk.

Controls:

- public release hygiene checks ignored internal paths and public doc references;
- staged secret scan is required before release PR commits;
- `cargo deny check licenses bans sources` is required;
- release-context `npm audit --omit=dev` is required and exceptions must be
  classified;
- detailed private QA reports stay under ignored report roots.

## Update And Signing Path

Surface: Windows installer, release workflow, signing credentials, and optional
future updater artifacts.

Risks:

- unsigned installer causes trust warnings or tampering risk;
- signing credentials leak;
- updater artifacts are unsigned or mismatched;
- public release notes point users at stale artifacts.

Controls:

- desktop installer remains the primary v1.0 artifact;
- signing secrets must stay in GitHub Actions secrets or the selected signing
  provider, never in the repository;
- updater remains deferred unless signed updater artifacts are added;
- release verification includes checksums, artifact names, signature behavior,
  and privacy links.

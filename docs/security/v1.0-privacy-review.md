# Nahou v1.0 Privacy Review

Last updated: 2026-06-25

This review records the v1.0 privacy posture for desktop, browser extension,
Office add-ins, local API, and optional local LLM use.

## Data Processed

- Desktop companion: user-selected text, clipboard text during capture/restore,
  capture method, source app name, and non-content error categories.
- UIA capture: selected text exposed by supported Windows UI Automation
  providers.
- Browser extension: active editable-field text from supported web editors,
  local API URL, writing mode, enabled or paused state, and disabled-site
  hostnames.
- Office add-ins: selected Word or PowerPoint text where Office APIs expose a
  safe selection flow.
- Local API: submitted text and deterministic suggestions in memory.
- Optional local LLM: submitted text sent to the configured loopback local model
  runtime after user configuration and consent.

## Where Data Is Stored

Nahou stores settings needed to operate the product. Examples include local API
URL, writing mode, extension enabled or paused state, disabled-site hostnames,
desktop capture preference, and local runtime configuration.

Nahou does not store raw selected text, active editor contents, source document
contents, suggestions, browsing history, credentials, screenshots, or raw API
responses by default.

## When Data Leaves Nahou

The default deterministic path stays on the user's machine and does not send text
to Nahou-hosted services.

Text may leave the immediate Nahou process only in these user-controlled local
cases:

- browser extension sends active-field text to the configured loopback Nahou API;
- Office add-ins send selected text to the configured local Nahou API;
- optional LLM suggestions send submitted text to the user's configured loopback
  local model runtime.

Nahou v1.0 has no hosted writing service, hosted telemetry, hosted analytics,
account sync, or operator review path.

## Default Retention

Default retention for raw selected text is none. Text is analyzed in memory and
discarded after the request/review flow. Settings are retained until the user
changes or removes them.

## Logs

Logs and diagnostics may include method names, app names, endpoint categories,
connection state, error categories, and package validation results. They must not
include raw selected text, source documents, credentials, access tokens, local
secret values, or screenshots containing private user text.

The public-release hygiene gate scans source-controlled log/report artifact
locations for obvious selected-text samples. Manual review is still required for
private QA screenshots and temporary local logs before any release artifact is
published.

## Reports

Public-safe reports should summarize pass/fail status, tested surfaces, artifact
hashes, and non-content error categories. Detailed manual QA evidence and
screenshots remain private and ignored under report roots. Public reports must
not include raw selected text.

## Screenshots

Store and release screenshots must use synthetic public-safe text. Screenshots of
private user writing, customer content, real documents, credentials, keys, or
tokens must not be committed or published.

## Extension Permissions

The browser extension uses the minimum permissions needed for the v1.0 local
editing flow. It stores settings in extension storage, injects content scripts to
detect supported editable fields, and connects only to configured loopback local
API URLs. Page-origin content-script messages cannot override stored API URL or
writing mode.

Sensitive-field exclusion is best effort. The extension avoids password, payment,
token, API-key, one-time-code, read-only, disabled, hidden, and
sensitive-looking fields where browser/editor metadata exposes enough
information to identify them. This does not mean every sensitive field on every
website is technically inaccessible.

## Office Permissions

The Office add-ins use document APIs to read selected Word or PowerPoint text and
write guarded replacements where supported. The production manifest must use
public HTTPS privacy/support URLs and must not claim hosts, requirement sets, or
platform behavior beyond tested release evidence.

Office sensitive-field exclusion is limited by Office host APIs and document
structure. Public wording must describe privacy controls and unsupported states
without claiming that all sensitive document content is impossible to access.

## Manual Log Review

Before release, the release owner must review local QA logs, generated package
manifests, public reports, and screenshot directories for raw selected text. Any
hit must be removed from public artifacts and either moved to an ignored private
report location or regenerated with synthetic public-safe text.

Current Phase 8 status: source-controlled public hygiene is automated; private
runtime logs and screenshots still require manual review during the release
candidate phase.
